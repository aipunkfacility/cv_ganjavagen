# generate-professional-readme

Этот навык предназначен для генерации исчерпывающей и визуально привлекательной документации `README.md` для проектов любого типа.

## Когда использовать

- При создании нового репозитория.
- При рефакторинге документации существующего проекта.
- Когда пользователь просит "сделать красивый ридми" или "описать проект".

## Структура идеального README

Следуйте этой структуре, если пользователь не задал иное:

1. **Заголовок**: Название проекта + короткое (1-2 предложения) описание.
2. **Бейджи**: Статус сборки, лицензия, версия, техи (по возможности).
3. **О проекте (About)**:
    - Подробное описание назначения.
    - Скриншоты или GIF-демо (если есть возможность сгенерировать или попросить).
    - Ключевые особенности (Features).
4. **Технологический стек (Built With)**: Список фреймворков, библиотек и инструментов.
5. **Начало работы (Getting Started)**:
    - Требования (Prerequisites).
    - Установка (Installation).
6. **Использование (Usage)**: Примеры кода, команд или сценариев использования.
7. **Структура проекта (Project Structure)**: Дерево файлов (опционально, но полезно).
8. **Роадмап (Roadmap)**: Планируемые функции.
9. **Вклад (Contributing)**: Как помочь проекту.
10. **Лицензия (License)**.
11. **Контакты (Contact)**.
12. **Благодарности (Acknowledgments)**.

## Принципы

1. **Visual First**: Используйте таблицу для выравнивания логотипов или скриншотов, если это уместно. Бейджи обязательны.
2. **Concise & Clear**: Пишите лаконично, разбивайте текст на списки.
3. **Code Blocks**: Все команды и примеры кода должны быть в блоках с подсветкой синтаксиса.
4. **Links**: Используйте относительные ссылки для файлов внутри репозитория и абсолютные для внешних ресурсов.
5. **Language**: По умолчанию генерируйте README на **русском языке**, если в запросе или коде проекта (например, в i18n) явно не доминирует другой язык.

## Workflow генерации

1. **Анализ**:
    - Прочитать `package.json` (или аналоги) для определения стека.
    - Прочитать существующий `README.md` (если есть) для сохранения контекста.
    - Изучить структуру файлов, чтобы понять архитектуру.
2. **Генерация**:
    - Создать структуру по шаблону выше.
    - Заполнить разделы на основе анализа кода.
    - Если информации не хватает (например, о лицензии или roadmap) — оставить placeholders или спросить пользователя, если это критично.
    - **Важно**: Если есть изображения в `src/assets` или `public`, попробуйте включить их в раздел скриншотов.
